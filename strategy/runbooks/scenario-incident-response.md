# 🚨 操作手册：事件响应

> **模式**：NEXUS-Micro | **周期**：数分钟到数小时 | **智能体**：3-8 个

---

## 场景

生产环境出了问题。用户受到影响。响应速度很重要，但正确处理同样重要。本操作手册涵盖从检测到事后分析的完整流程。

## 严重程度分类

| 级别 | 定义 | 示例 | 响应时间 |
|-------|-----------|----------|--------------|
| **P0 — 严重** | 服务完全中断、数据丢失、安全漏洞 | 数据库损坏、DDoS 攻击、认证系统故障 | 立即（全员出动） |
| **P1 — 高** | 主要功能故障、严重性能下降 | 支付处理中断、50%+ 错误率、10 倍延迟 | < 1 小时 |
| **P2 — 中** | 次要功能故障、有临时解决方案 | 搜索不可用、非关键 API 错误 | < 4 小时 |
| **P3 — 低** | 外观问题、轻微不便 | 样式 bug、拼写错误、轻微 UI 故障 | 下个冲刺 |

## 按严重程度的响应团队

### P0 — 严重响应团队
| 智能体 | 角色 | 行动 |
|-------|------|--------|
| **基础设施维护师** | 事件指挥官 | 评估范围、协调响应 |
| **DevOps 自动化师** | 部署/回滚 | 执行回滚（如需要） |
| **后端架构师** | 根因调查 | 诊断系统问题 |
| **前端开发者** | UI 侧调查 | 诊断客户端问题 |
| **支持响应师** | 用户沟通 | 状态页面更新、用户通知 |
| **执行摘要生成师** | 利益相关者沟通 | 实时执行更新 |

### P1 — 高响应团队
| 智能体 | 角色 |
|-------|------|
| **基础设施维护师** | 事件指挥官 |
| **DevOps 自动化师** | 部署支持 |
| **相关开发者智能体** | 修复实现 |
| **支持响应师** | 用户沟通 |

### P2 — 中响应
| 智能体 | 角色 |
|-------|------|
| **相关开发者智能体** | 修复实现 |
| **证据收集师** | 验证修复 |

### P3 — 低响应
| 智能体 | 角色 |
|-------|------|
| **冲刺优先级师** | 添加到待办事项 |

## 事件响应序列

### 步骤 1：检测与分诊（0-5 分钟）

```
触发：监控警报 / 用户报告 / 智能体检测

基础设施维护师：
1. 确认警报
2. 评估范围和影响
   - 多少用户受影响？
   - 哪些服务受到影响？
   - 数据是否有风险？
3. 分类严重程度（P0/P1/P2/P3）
4. 激活适当的响应团队
5. 创建事件频道/线程

输出：事件分类 + 响应团队激活
```

### 步骤 2：调查（5-30 分钟）

```
并行调查：

基础设施维护师：
├── 检查系统指标（CPU、内存、网络、磁盘）
├── 查看错误日志
├── 检查最近的部署
└── 验证外部依赖

后端架构师（如 P0/P1）：
├── 检查数据库健康状态
├── 查看 API 错误率
├── 检查服务通信
└── 识别故障组件

DevOps 自动化师：
├── 查看最近的部署历史
├── 检查 CI/CD 流水线状态
├── 准备回滚（如需要）
└── 验证基础设施状态

输出：根因已识别（或缩小到组件）
```

### 步骤 3：缓解（15-60 分钟）

```
决策树：

如果由最近部署引起：
  → DevOps 自动化师：执行回滚
  → 基础设施维护师：验证恢复
  → 证据收集师：确认修复

如果由基础设施问题引起：
  → 基础设施维护师：扩容/重启/故障转移
  → DevOps 自动化师：支持基础设施变更
  → 验证恢复

如果由代码 bug 引起：
  → 相关开发者智能体：实现热修复
  → 证据收集师：验证修复
  → DevOps 自动化师：部署热修复
  → 基础设施维护师：监控恢复

如果由外部依赖引起：
  → 基础设施维护师：激活备用/缓存
  → 支持响应师：向用户沟通
  → 监控外部恢复

全程：
  → 支持响应师：每 15 分钟更新状态页面
  → 执行摘要生成师：向利益相关者简报（仅 P0）
```

### 步骤 4：解决验证（修复后）

```
证据收集师：
1. 验证修复是否解决问题
2. 截图证明工作状态
3. 确认没有引入新问题

基础设施维护师：
1. 验证所有指标恢复正常
2. 确认没有级联故障
3. 修复后监控 30 分钟

API 测试师（如与 API 相关）：
1. 在受影响的端点上运行回归测试
2. 验证响应时间正常化
3. 确认错误率回到基线

输出：事件解决确认
```

### 步骤 5：事后分析（48 小时内）

```
工作流优化师主导事后分析：

1. 时间线重建
   - 问题何时引入？
   - 何时检测到？
   - 何时解决？
   - 用户影响总时长

2. 根因分析
   - 什么失败了？
   - 为什么失败？
   - 为什么没有更早发现？
   - 5 个为什么分析

3. 影响评估
   - 受影响用户
   - 收入影响
   - 声誉影响
   - 数据影响

4. 预防措施
   - 什么监控可以更早发现这个问题？
   - 什么测试可以预防这个问题？
   - 需要什么流程变更？
   - 需要什么基础设施变更？

5. 行动项
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]

输出：事后分析报告 → 冲刺优先级师将预防任务添加到待办事项
```

## 沟通模板

### 状态页面更新（支持响应师）
```
[时间戳] — [服务名称] 事件

状态：[调查中 / 已识别 / 监控中 / 已解决]
影响：[用户影响描述]
当前行动：[我们正在做什么]
下次更新：[预计下次更新时间]
```

### 执行更新（执行摘要生成师 — 仅 P0）
```
事件简报 — [时间戳]

情况：[服务] [中断/降级]，影响 [N 用户/% 流量]
原因：[已知/调查中] — [如已知则简要描述]
行动：[正在做什么] — 预计 [时间估计]
影响：[业务影响 — 收入、用户、声誉]
下次更新：[时间戳]
```

## 升级矩阵

| 条件 | 升级到 | 行动 |
|-----------|------------|--------|
| P0 在 30 分钟内未解决 | 工作室制作人 | 额外资源、供应商升级 |
| P1 在 2 小时内未解决 | 项目协调员 | 资源重新分配 |
| 疑似数据泄露 | 法律合规检查师 | 监管通知评估 |
| 用户数据受影响 | 法律合规检查师 + 执行摘要生成师 | GDPR/CCPA 通知 |
| 收入影响 > $X | 财务追踪师 + 工作室制作人 | 业务影响评估 |
